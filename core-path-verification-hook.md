# 核心情境驗收 Hook（全域規則 + 可執行 Hook）

## 背景

這條規則原本只存在「使用者口頭提醒」跟「Claude記憶」裡，這兩者對 Claude Code 完全沒有約束力——記憶只在對話裡有效，下一個 session、下一次重開，這個提醒就消失了。這次直接做成 hook，讓它變成 Claude Code 自己的環境會主動攔下來的東西，不依賴任何人記得提醒。

## 規則本身

任何時候要回報「已完成」或「結案」之前，必須先確認：**核心使用情境（不是邊界情況，是最基本、最常見、不帶任何選填參數的那個用法）有沒有至少端到端跑過一次**。不能只看單元測試通過、不能只看某個被特別要求驗證的子功能測過。

具體案例（這條規則因此而生）：LEAPS 功能的 Stage 1 自動偵測路徑（不輸入履約價，純粹依 Delta>=0.80 自動篩選候選），是整個功能最基本、預設的使用方式。但開發過程中所有的端到端測試都是帶著 `user_strike`（選填的手動覆寫參數）跑的，導致這條最基本路徑從未被真正驗證過，卻已經被回報「結案」。

## Hook 實作

在 commit 或任務結束前的 pre-completion 檢查點，加入這個檢查：

```bash
# .claude/hooks/pre-completion-core-path-check.sh（示意，依專案實際 hook 框架調整）
# 觸發時機：任何「完成」「結案」「驗收通過」字樣出現在 commit message 或任務總結時

SUMMARY_TEXT="$1"  # 假設 hook 框架會傳入 commit message 或任務摘要文字

if echo "$SUMMARY_TEXT" | grep -qiE "結案|已完成|驗收通過|done|completed"; then
  echo "⚠️  偵測到完成/結案宣告。在繼續前，請明確回答："
  echo ""
  echo "  1. 這個功能/修復最基本、不帶任何選填參數的核心使用情境，"
  echo "     是否至少端到端跑過一次（不是單元測試，是真實觸發一次）？"
  echo "  2. 如果只測過帶特定參數（例如選填覆寫值）的情境，"
  echo "     沒測過完全預設、不帶任何參數的情境，這不算完成。"
  echo ""
  echo "若以上第1點答案不是「是，已驗證」，不要在這次回報中使用「完成/結案」字樣。"
fi
```

（實際擋下/僅警告、用什麼方式判斷「核心情境是否測過」，依專案現有 hook 框架的能力調整；最低限度至少要做到「每次完成宣告前強制跳出提醒」，理想狀態是能要求填寫具體的核心情境驗證證據才放行。）

## 補充：併入 CLAUDE.md 的版本（hook 觸發不到的情況下的最後防線）

如果這次的 hook 框架沒辦法精確抓到「完成宣告」這個時機點，至少把下面這段寫進 CLAUDE.md 的開頭區域（跟其他全域規則放在一起，不要埋在文件中段容易被忽略的位置）：

> **任何回報「已完成」「結案」「驗收通過」之前，先自問：核心使用情境（最基本、不帶任何選填參數的那個用法）有沒有至少端到端跑過一次？不能只看單元測試通過。如果答案不確定，先去跑一次，不要先回報完成。**

## 驗收標準

- [ ] Hook 已安裝並能在涉及「完成/結案」宣告時觸發提醒（或更強的攔截）。
- [ ] CLAUDE.md 開頭區域有這條規則的明文版本，作為 hook 失效時的備援。
- [ ] 之後任何「結案」回報，都附上「核心情境已端到端驗證」的具體證據（不是只有單元測試數字）。
